# СПЕЦИФИКАЦИЯ ПРОЕКТА: GenBI — Генеративная аналитическая платформа

## 1. Обзор проекта

**Контекст:** Крупнейшая алюминиевая компания мира. Подразделение обеспечения сырья производства (алюминий + энергоресурсы).

**Проблема:** Сотрудники тратят часы на ручной сбор данных из SAP, котировок, курсов и новостей. Запросы к аналитике требуют участия специалистов по данным. Нет единого интерфейса для быстрого получения инсайтов.

**Решение:** Веб-платформа с генеративным ИИ-интерфейсом, где сотрудник формулирует вопрос простым языком и получает:
- Текстовый ответ с анализом
- Интерактивные графики и визуализации
- Числовые показатели и расчёты
- Резюме/выводы

**Целевая аудитория:** 50 пользователей смешанных ролей (аналитики, руководители, закупщики).

## 2. Функциональные требования

### 2.1. Генеративный интерфейс (Чат)

| Требование | Описание |
|---|---|
| Естественный язык | Пользователь вводит вопрос на русском/английском |
| Контекст сессии | ИИ помнит контекст диалога в рамках сессии |
| История запросов | Сохранение истории с возможностью вернуться к прошлым запросам |
| Типы ответов | Текст, таблицы, графики, расчёты, смешанные |
| Уточняющие вопросы | ИИ может уточнить параметры запроса (период, актив, источник) |

### 2.2. Типы аналитических запросов

**Категория 1 — Исторический анализ котировок:**
> «Как себя вела котировка алюминия на LME за январь-март 2025?»
> «Покажи динамику курса доллара к рублю за последний квартал»

**Категория 2 — Многофакторный анализ:**
> «Какие макрофакторы могли повлиять на курс в период X?»
> «Сравни динамику алюминия и цен на энергию за полгода»

**Категория 3 — Расчёты и модели:**
> «Посчитай формулу цены на основе котировок и курса»
> «Рассчитай стоимость закупки глинозёма с учётом логистики»

**Категория 4 — Сводки и мониторинг:**
> «Что произошло на рынке за последнюю неделю?»
> «Есть ли отклонения от плана по закупкам?»

**Категория 5 — Прогнозирование:**
> «Спрогнозируй котировку на следующий месяц»
> «Какой объём закупок потребуется в Q3?»

### 2.3. Визуализации

| Тип | Описание |
|---|---|
| Линейные графики | Динамика котировок, курсов, объёмов |
| Столбчатые диаграммы | Сравнение периодов, категорий |
| Тепловые карты | Корреляция факторов |
| Scatter plots | Зависимости между показателями |
| Candlestick | Котировки (как на бирже) |
| Таблицы | Детализированные данные |
| KPI-карточки | Ключевые показатели (текущее значение, Δ, тренд) |

### 2.4. Авторизация

- Простая авторизация: логин/пароль
- Ролевая модель (базовая): пользователь / администратор / модератор
- Аудит: кто, когда, что запрашивал (логирование)

### 2.5. Форма обратной связи (Feedback)

Цель — собирать оценки качества ответов ИИ для дообучения и улучшения.

#### 2.5.1. Типы feedback

| Тип | Описание | Визуал |
|---|---|---|
| **Лайк/Дизлайк** | Быстрая оценка: «Ответ полезен» / «Не полезен» | 👍 / 👎 под каждым ответом |
| **Категория проблемы** | Если дизлайк — выбор причины | Выпадающий список |
| **Текстовый комментарий** | Пользователь пишет, что не так / что можно улучшить | Поле ввода (опционально) |
| **Исправление ответа** | Пользователь предлагает правильный ответ | Поле для редактирования |
| **Оценка точности** | «Данные верны» / «Данные некорректны» | Чекбокс |

#### 2.5.2. Категории проблем (при дизлайке)

```
- Некорректные данные
- Неполный ответ
- Неверная интерпретация запроса
- Ошибка в расчётах
- График не отображается
- Ответ не на русском языке
- Другое (с указанием)
```

#### 2.5.4. Правила обработки feedback

| Правило | Описание |
|---|---|
| Анонимность | Feedback привязан к пользователю, но виден только администраторам |
| Невозможно удалить | Feedback нельзя удалить пользователям (только модератору) |
| Валидация | Комментарий ≥ 10 символов, исправление ≥ 5 символов |
| Дедупликация | Один feedback на сообщение (можно обновить) |
| Уведомления | Администратор получает уведомление о дизлайках |
| Экспорт | Feedback выгружается в CSV/Excel для анализа |

### 2.6. База знаний (Knowledge Base)

Система хранения проверенной информации для улучшения качества ответов ИИ и защиты от галлюцинаций.

#### 2.6.1. Архитектура Knowledge Base

```
┌─────────────────────────────────────────────────────────────────┐
│                     Knowledge Base Pipeline                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Источники                 Обработка              Хранение      │
│  ─────────                 ─────────              ─────────     │
│  • Документы (PDF, Word)   → Парсинг             → Greenplum   │
│  • FAQ (руководства)       → Чанкинг              → Векторный   │
│  • Нормативные акты        → Эмбеддинги          индекс        │
│  • Справочники             → Валидация            (pgvector)    │
│  • История feedback        → Версионирование                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.6.2. Типы документов в базе знаний

| Тип | Описание | Приоритет |
|---|---|---|
| **Нормативные документы** | ГОСТы, ТУ, регламенты | Высокий (неизменяемые) |
| **Справочники** | Коды металлов, единицы измерения, формулы | Высокий |
| **FAQ** | Типовые вопросы и ответы | Средний (обновляемые) |
| **Исторические данные** | Проверенные ответы с feedback | Средний |
| **Инструкции** | Как читать котировки, как рассчитывать | Средний |
| **Внешние данные** | Новости, аналитика от экспертов | Низкий (временные) |

#### 2.6.3. Защита от искажений (Anti-Hallucination)

| Механизм | Реализация |
|---|---|
| **RAG (Retrieval-Augmented Generation)** | LLM получает релевантные документы из KB перед генерацией ответа |
| **Ссылки на источники** | Каждый ответ содержит ссылки на документы из KB |
| **Уровень доверия** | Показывается % уверенности в ответе (на основе релевантности) |
| **Валидация данных** | Проверка: данные из ответа должны совпадать с DWH |
| **Человек-в-цикле** | Критические ответы требуют подтверждения модератора |
| **Версионирование** | Каждый документ имеет версию и дату обновления |
| **Логирование** | Все обращения к KB логируются для аудита |

#### 2.6.5. Модерация контента

| Проверка | Описание | Действие при нарушении |
|---|---|---|
| **Спам** | Документ содержит рекламу/спам | Отклонить, уведомить admin |
| **Дубликат** | Документ уже существует | Предложить обновление |
| **Размер** | Файл > 50MB | Отклонить с ошибкой |
| **Формат** | Неподдерживаемый формат | Отклонить с ошибкой |
| **Мусор** | Пустой/бессмысленный текст | Отклонить |
| **Версионность** | Устаревший документ | Пометить как deprecated |
| **Права доступа** | Документ требует грифа | Ограничить доступ |

```python
    class Config:
        env_file = ".env"

# app/core/database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from app.core.config import settings

# Greenplum connection (PostgreSQL-compatible)
DATABASE_URL = f"postgresql://{settings.GP_USER}:{settings.GP_PASSWORD}@{settings.GP_HOST}:{settings.GP_PORT}/{settings.GP_DATABASE}"

engine = create_engine(
    DATABASE_URL,
    pool_size=20,
    max_overflow=10,
    pool_pre_ping=True,
    connect_args={
        "connect_timeout": 10,

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

### 3.2.2. Модели (LLM + Embeddings)

#### Основная LLM-модель (Генерация ответов)

| Параметр | Значение | Описание |
|---|---|---|
| **Провайдер** | LiteLLM (OpenAI Compatible) | Проксирует запросы к LLM |
| **Модель (рекомендуемая)** | GPT-4o / GPT-4-turbo | Для генерации SQL и аналитических ответов |
| **Модель (альтернатива)** | Claude 3.5 Sonnet / Gemini 1.5 Pro | Конкурентные модели через LiteLLM |
| **Модель (эконом)** | GPT-4o-mini / GPT-3.5-turbo | Для простых запросов, парсинга |
| **Температура** | 0.1 - 0.3 | Низкая для точных SQL-запросов |
| **Max tokens** | 4096 | Для длинных ответов с графиками |
| **Контекстное окно** | 128K токенов | Для больших datasets в контексте |

```python
# Конфигурация моделей в .env
LLM_PRIMARY_MODEL = "gpt-4o"              # Основная модель
LLM_FAST_MODEL = "gpt-4o-mini"            # Быстрая модель (парсинг, простые запросы)
LLM_TEMPERATURE = 0.2                     # Низкая температура для точности
```

#### Детали Clarification Flow

**Когда ИИ задаёт уточняющие вопросы:**

| Ситуация | Пример вопроса |
|---|---|
| Не указан период | «За какой период показать данные?» |
| Не указан металл | «Какой металл: алюминий, медь, цинк?» |
| Неоднозначный запрос | «Вы хотите сравнение или детальный анализ?» |
| Нет единиц измерения | «В каких единицах: USD/тонна, RUB/тонна?» |
| Сложный расчёт | «Уточните формулу: какая переменная X?» |

**Когда ИИ НЕ задаёт вопрос (генерирует сразу):**

| Ситуация | Действие |
|---|---|
| Запрос полный и однозначный | Сразу генерация SQL |
| Есть контекст из истории сессии | Используем предыдущие уточнения |
| Простой справочный запрос | «Что такое LME?» — сразу ответ |

**Ограничения clarification:**

| Параметр | Значение |
|---|---|
| Макс. уточняющих вопросов за раз | 3 |
| Макс. раундов уточнения | 2 (после 2 — генерация с лучшими предположениями) |
| Модель для уточнений | gpt-4o-mini (быстрая, дешёвая) |
| Таймаут ожидания ответа | 5 минут (потом — генерация с defaults) |

## 5. Модель данных (Greenplum)

> **Примечание:** Greenplum — MPP-виарехаус. Ключевые особенности:
> - Таблицы требуют `DISTRIBUTED BY` (ключ распределения)
> - Нет foreign key ограничений (проверяются на уровне приложения)
> - Рекомендуется columnar storage для аналитических таблиц (`WITH (appendoptimized=true, columnar=true)`)
> - SERIAL не поддерживается — используются IDENTITY или SEQUENCE

```sql
-- =====================================================
-- Пользователи (OLTP-таблица, row storage)
-- =====================================================
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(20) DEFAULT 'user',
    created_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (id);

-- =====================================================
-- Сессии чата (OLTP, row storage)
-- =====================================================
CREATE TABLE chat_sessions (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(255),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (user_id);

-- =====================================================
-- Сообщения (OLTP, row storage)
-- =====================================================
CREATE TABLE chat_messages (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    session_id BIGINT NOT NULL,
    role VARCHAR(10) NOT NULL,           -- 'user' или 'assistant'
    message_type VARCHAR(20) DEFAULT 'text', -- 'text', 'clarification', 'chart', 'table'
    content TEXT NOT NULL,
    metadata JSONB,                      -- графики, таблицы, SQL, clarification_options
    created_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (session_id);

-- =====================================================
-- Кэш запросов к LLM (OLTP, row storage)
-- =====================================================
CREATE TABLE query_cache (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    query_hash VARCHAR(64) UNIQUE NOT NULL,
    response JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP
) DISTRIBUTED BY (query_hash);

-- =====================================================
-- Лог аудита (append-only, columnar)
-- =====================================================
CREATE TABLE audit_log (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    user_id BIGINT,
    action VARCHAR(50),
    details JSONB,
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (user_id);

-- =====================================================
-- Котировки (аналитические данные, columnar)
-- =====================================================
CREATE TABLE quotes (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    quote_date DATE NOT NULL,
    metal VARCHAR(50) NOT NULL,        -- AL, CU, ZN, etc.
    exchange VARCHAR(20) NOT NULL,     -- LME, SHFE
    close_price NUMERIC(12,2),
    volume BIGINT,
    change_pct NUMERIC(6,2),
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (metal)
PARTITION BY RANGE (quote_date) (
    START ('2020-01-01') INCLUSIVE
    END ('2030-01-01') EXCLUSIVE
    EVERY (INTERVAL '1 year')
);

-- =====================================================
-- Курсы валют (columnar)
-- =====================================================
CREATE TABLE currencies (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    rate_date DATE NOT NULL,
    currency_pair VARCHAR(10) NOT NULL, -- USD/RUB, EUR/RUB
    rate NUMERIC(10,4),
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (currency_pair)
PARTITION BY RANGE (rate_date) (
    START ('2020-01-01') INCLUSIVE
    END ('2030-01-01') EXCLUSIVE
    EVERY (INTERVAL '1 year')
);

-- =====================================================
-- Товары/коммодити (columnar)
-- =====================================================
CREATE TABLE commodities (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    commodity_date DATE NOT NULL,
    name VARCHAR(100) NOT NULL,        -- бокситы, глинозём, анодная масса
    price NUMERIC(12,2),
    unit VARCHAR(20),                  -- USD/t, RUB/t
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (name)
PARTITION BY RANGE (commodity_date) (
    START ('2020-01-01') INCLUSIVE
    END ('2030-01-01') EXCLUSIVE
    EVERY (INTERVAL '1 year')
);

-- =====================================================
-- Новости (columnar)
-- =====================================================
CREATE TABLE news (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    news_date TIMESTAMP NOT NULL,
    title TEXT NOT NULL,
    source VARCHAR(100),
    category VARCHAR(50),              -- macro, industry, company
    impact VARCHAR(20),                -- positive, negative, neutral
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (source)
PARTITION BY RANGE (news_date) (
    START ('2020-01-01') INCLUSIVE
    END ('2030-01-01') EXCLUSIVE
    EVERY (INTERVAL '1 year')
);

-- =====================================================
-- Поставщики (row storage, справочник)
-- =====================================================
CREATE TABLE suppliers (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    country VARCHAR(100),
    product VARCHAR(100),              -- бокситы, глинозём
    conditions TEXT,
    created_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (id);

-- =====================================================
-- Закупки (columnar)
-- =====================================================
CREATE TABLE purchases (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    purchase_date DATE NOT NULL,
    product VARCHAR(100) NOT NULL,
    supplier_id BIGINT,
    quantity NUMERIC(12,2),
    price_per_unit NUMERIC(12,2),
    total_amount NUMERIC(14,2),
    currency VARCHAR(10) DEFAULT 'USD',
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (product)
PARTITION BY RANGE (purchase_date) (
    START ('2020-01-01') INCLUSIVE
    END ('2030-01-01') EXCLUSIVE
    EVERY (INTERVAL '1 year')
);

-- =====================================================
-- Feedback на ответы (OLTP, row storage)
-- =====================================================
CREATE TABLE feedback (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    message_id BIGINT NOT NULL,          -- ID сообщения ассистента
    user_id BIGINT NOT NULL,             -- Кто оставил feedback
    rating VARCHAR(10) NOT NULL,         -- 'like' или 'dislike'
    category VARCHAR(50),                -- Категория проблемы (при дизлайке)
    comment TEXT,                        -- Текстовый комментарий
    corrected_answer TEXT,               -- Исправленный ответ пользователя
    is_data_accurate BOOLEAN,            -- Верны ли данные
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (user_id);

-- =====================================================
-- Документы базы знаний (OLTP, row storage)
-- =====================================================
CREATE TABLE kb_documents (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title VARCHAR(500) NOT NULL,
    doc_type VARCHAR(50) NOT NULL,       -- normative, faq, instruction, reference, news
    file_path VARCHAR(1000),             -- Путь к файлу на диске
    file_hash VARCHAR(64),               -- Хеш файла для дедупликации
    version INT DEFAULT 1,
    status VARCHAR(20) DEFAULT 'active', -- active, deprecated, pending
    chunk_count INT DEFAULT 0,           -- Количество чанков
    uploaded_by BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
) DISTRIBUTED BY (id);

-- =====================================================
-- Чанки документов (columnar для быстрого поиска)
-- =====================================================
CREATE TABLE kb_chunks (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    document_id BIGINT NOT NULL,
    chunk_index INT NOT NULL,            -- Порядковый номер чанка
    content TEXT NOT NULL,               -- Текст чанка
    token_count INT,                    -- Количество токенов
    embedding VECTOR(1536),             -- Эмбеддинг (pgvector ext.)
    metadata JSONB,                     -- Доп. данные (страница, заголовок)
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (document_id);

-- =====================================================
-- История изменений KB (append-only, columnar)
-- =====================================================
CREATE TABLE kb_audit_log (
    id BIGINT GENERATED ALWAYS AS IDENTITY,
    document_id BIGINT NOT NULL,
    action VARCHAR(50) NOT NULL,         -- create, update, delete, publish
    old_version INT,
    new_version INT,
    changed_by BIGINT NOT NULL,
    details JSONB,
    created_at TIMESTAMP DEFAULT NOW()
) WITH (appendoptimized=true, columnar=true)
DISTRIBUTED BY (document_id);

-- =====================================================
-- Индексы для быстрого поиска
-- =====================================================
-- Индекс для поиска по эмбеддингам (cosine similarity)
CREATE INDEX idx_kb_chunks_embedding ON kb_chunks 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- Индекс для поиска по документам
CREATE INDEX idx_kb_documents_status ON kb_documents(status);
CREATE INDEX idx_kb_documents_type ON kb_documents(doc_type);

-- Индекс для feedback
CREATE INDEX idx_feedback_message ON feedback(message_id);
CREATE INDEX idx_feedback_rating ON feedback(rating);
```

## 6.1. Особенности работы с Greenplum

| Особенность | Рекомендация |
|---|---|
| **Distribution Keys** | Выбирать по колонке, по которой чаще всего фильтруются/джойнятся данные |
| **Columnar Storage** | Использовать для аналитических таблиц (quotes, currencies, commodities) |
| **Row Storage** | Использовать для OLTP-таблиц (users, sessions, messages) |
| **Partitioning** | Партиционировать по дате для больших таблиц |
| **No Foreign Keys** | Целостность данных проверять на уровне приложения |
| **No SERIAL** | Использовать `GENERATED ALWAYS AS IDENTITY` |
| **JSONB** | Поддерживается в Greenplum 6+, можно использовать для метаданных |
| **Query Optimization** | Использовать `EXPLAIN ANALYZE` для оптимизации запросов |
| **Statistics** | Регулярно обновлять статистику: `ANALYZE table_name;` |
